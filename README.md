# URR (우르르)

> K-POP 찐팬을 위한 공정 티켓팅 플랫폼

매크로·봇 차단, 멤버십 등급제로 팬 활동을 정량화해 티켓 우선권 부여. 티켓팅 → 양도 → 커뮤니티를 하나의 생태계로 연결합니다.

## 프로젝트 개요

### 해결하는 문제

| 기존 문제                 | URR의 접근                            |
| ------------------------- | ------------------------------------- |
| 매크로·봇이 대기열 점령   | VQA(팬 인증 시각 질문)로 봇 자동 차단 |
| 찐팬도 티켓 구하기 어려움 | 멤버십 등급별 선예매 우선순위         |
| 암표·외부 양도 난무       | 플랫폼 내 공식 양도 마켓으로 한정     |
| 티켓팅·양도·커뮤니티 분산 | 하나의 플랫폼에서 전부 해결           |

### 멤버십 등급 체계

팬 활동을 정량화해 더 열심히 응원할수록 더 많은 혜택을 제공합니다.

| 등급           | 조건             |   선예매 순서   | 양도 수수료 |
| -------------- | ---------------- | :-------------: | :---------: |
| Lv.4 다이아 🌩️ | 팬 점수 달성     |    **1순위**    |     5%      |
| Lv.3 골드 ⚡   | 팬 점수 달성     | **2순위** (+1h) |     5%      |
| Lv.2 실버 ☁️   | 멤버십 가입 즉시 |    **3순위**    |     10%     |
| Lv.1 브론즈 🌫️ | 회원가입         |   일반 예매만   |  양도 불가  |

## 아키텍처

### Feature-Sliced Design (FSD)

레이어 간 단방향 의존성을 강제해 기능 추가/삭제 시 사이드 이펙트를 최소화합니다.

```
src/
├── app/        # Next.js App Router — 라우팅 진입점만. 로직 없음.
├── widgets/    # 페이지를 구성하는 큰 UI 블록 (여러 feature 조합)
├── features/   # 단일 사용자 행동 단위 (로그인, 예매, 결제 등)
├── entities/   # 도메인 모델 + 관련 UI (User, Artist, Event, Ticket)
└── shared/
    ├── api/    # fetch 클라이언트, 공통 에러 핸들링
    ├── lib/    # format.ts, constants.ts, utils.ts
    └── ui/     # shadcn 컴포넌트 (button, input, dialog …)
```

**레이어 규칙**: `app → widgets → features → entities → shared` 방향만 import 허용.

### feature 내부 구조

```
features/<domain>/<feature-name>/
├── ui/       # React 컴포넌트 (JSX + 스타일)
├── model/    # 상태·로직 훅 (useXxx.ts)
├── api/      # fetch 함수
└── index.ts  # barrel export
```

## 핵심 기능

### 예매 플로우 (5단계)

```
VQA 팬 인증 → 실시간 대기열 → 구역/좌석 선택 → 결제 → 예매 확인
```

- **VQA**: 아티스트 관련 시각 질문으로 매크로 차단. 정답 틀리면 예매 불가.
- **실시간 대기열**: 폴링 기반 순번 대기. 등급별 입장 시각 차등 적용.
- **좌석 선택**: SVG 기반 인터랙티브 좌석도. 등급·가격·잔여 상태 색상 표시.
- **결제**: Toss Payments 모킹.

### 양도 마켓

- `listed` 상태만 수정·취소 가능 (마이페이지 양도 탭)
- 수정: 가격 변경 → 수수료 5% 자동 계산 표시
- 취소: 확인 다이얼로그 → 상태 `cancelled` 전환

### 멤버십 가입 플로우 (4단계)

```
아티스트 선택 → 티어 소개 → 결제 → 완료
```

## 개발 방식 — AI 에이전트 협업

이 프로젝트에서 **프론트엔드는 1인 담당**으로, Claude Code(AI 에이전트)를 활용한 워크플로우로 진행됐습니다.

### 브랜치 전략

에이전트마다 독립 브랜치를 할당해 병렬 작업 후 PR로 통합합니다.

```
dev
 ├── feat/<scope>              ← 에이전트 A
 ├── feat/<scope>              ← 에이전트 B
 └── review/merge-<a>-<b>     ← 리뷰 에이전트 (충돌 조율)
          ↓ PR (Squash merge)
         dev
          ↓ PR (Merge commit)
         main
```

- **에이전트 1개 = 브랜치 1개**: 태스크 간 코드 의존 방지
- **리뷰 에이전트**: 두 feature 브랜치의 diff를 분석해 충돌 없는 병합 코드 자동 작성
- 자세한 규칙: [`CONTRIBUTING.md`](CONTRIBUTING.md)

## 트러블슈팅

### Claude Code 일일 사용량 한도 — 이틀 연속 작업 중단

**현상**

AI 에이전트를 활용해 여러 페이지를 병렬로 개발하던 중, 이틀 연속으로 Claude의 일일 사용량 한도(rate limit)에 도달해 에이전트가 작업 도중 강제 중단됐습니다. 특히 컨텍스트가 긴 세션(좌석 선택 SVG 구현, 예매 플로우 5단계 등)에서 반복적으로 발생했습니다.

**원인 분석**

- 큰 기능을 하나의 세션에서 처음부터 끝까지 구현하려다 보니 단일 세션의 토큰 소비가 과도하게 컸습니다.
- 작업 단위를 사전에 분리하지 않아 중단 시 재시작 비용(컨텍스트 복구)도 컸습니다.
- 여러 에이전트를 동시에 실행하면서 한도 소진 속도가 가속됐습니다.

**해결 방법**

| 변경 전                          | 변경 후                                              |
| -------------------------------- | ---------------------------------------------------- |
| 기능 전체를 하나의 세션에서 구현 | UI / 로직(hook) / API 레이어를 세션 단위로 분리      |
| 에이전트 무제한 병렬 실행        | 하루 최대 2개 에이전트 동시 실행으로 제한            |
| 컨텍스트 누적 방치               | `/compact` 명령으로 주기적 컨텍스트 압축             |
| 작업 도중 설계 변경              | 에이전트 시작 전 CLAUDE.md 기준으로 태스크 명세 완성 |

FSD 아키텍처의 레이어 분리 원칙이 이 문제 해결에 직접적으로 도움이 됐습니다. `ui/` → `model/` → `api/` 순으로 세션을 나눠도 각 레이어가 독립적이라 중단·재개가 자연스러웠습니다.

**재발 방지**

에이전트에게 태스크를 줄 때 아래 형식을 고정으로 사용합니다.

```
브랜치: feat/<scope>
기준: dev
작업 범위: [단일 레이어 또는 단일 컴포넌트로 한정]
완료 조건: npm run build 통과, PR 생성
```

범위가 불명확한 태스크는 에이전트를 시작하기 전에 직접 분해합니다.
